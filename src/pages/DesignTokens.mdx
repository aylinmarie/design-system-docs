## Overview

Design tokens are variables that store specific design choices. They hold data for attributes like colors, fonts, and spacing.

## Standard format

These variables use a common JSON format. This standard lets different software tools and codebases share the same visual information reliably.

The [W3C Design Tokens Community Group](https://www.w3.org/community/design-tokens/) is standardizing a JSON format for design tokens. Aligning to it makes your tokens portable between tools — Tokens Studio, Style Dictionary, and others already support reading and writing this format.

```json
{
  "color": {
    "interactive": {
      "primary": {
        "$value": "{global.color.blue.600}",
        "$type": "color",
        "$description": "Primary action color"
      }
    }
  }
}
```

<Callout type="info">
  The [W3C Design Tokens spec](https://tr.designtokens.org/format/) uses `$type` and `$value` as reserved keys, and references other tokens with curly brace syntax: `{token.name}`.
</Callout>

## Token levels

Teams often organize tokens into three categories:

**Primitives** — basic values, such as a hex code.

**Semantics** — names that describe a purpose, such as "action-color".

**Components** — values for a specific item, such as a button.

## Naming structure

A token name has three parts that explain its use:

**Foundation** — the type of attribute, like color.

**Property** — the UI part, like a border.

**Modifier** — the state or purpose, like "hover".

```
// Foundation: what it applies to
// Property: what visual attribute
// Modifier: the state or purpose

color.text.default          // default body text
color.text.muted            // secondary/helper text
color.border.interactive    // borders on interactive elements
color.border.interactive.hover  // hover state border
spacing.component.padding.sm    // small component padding
```

Avoid encoding values in names (`color-gray-200` is a primitive token, not a semantic one) and avoid encoding components in semantic names (`button-blue` leaks into the wrong tier).

## Themes and updates

Tokens make theming possible. A system has light and dark modes by changing the values assigned to the token names. When a design changes, a team updates the token once to change the style across the entire product.

## Tooling & delivery

The primary tools in this space:

- [**Style Dictionary**](https://styledictionary.com/) (Amazon) — transform tokens into CSS custom properties, Sass variables, iOS/Android formats. The most battle-tested tool.
- **Theo** (Salesforce) — similar to Style Dictionary, slightly different DX.
- **Token Pipeline** / **Token Transformer** — part of the [Tokens Studio](https://tokens.studio/) ecosystem, closer to the W3C spec format.
- [**Tokens Studio for Figma**](https://tokens.studio/) — design-side authoring that outputs W3C-format JSON, consumed by a build step.

The general architecture looks like this:

```
Design (Figma / Tokens Studio)
  ↓  JSON token files
Build step (Style Dictionary)
  ↓  Platform-specific outputs
├── CSS custom properties  (web)
├── Swift/ObjC constants  (iOS)
└── Kotlin constants       (Android)
```
