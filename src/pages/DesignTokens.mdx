## Overview

Design tokens are variables that store specific design choices. They hold data for attributes like colors, fonts, and spacing.

<DemoCard label="Global color tokens">
  {tokenSwatches.map((token) => (
    <Flex key={token.name} align="center" gap="4" className="demo-row">
      <Box
        className="color-swatch"
        style={{ "--swatch-bg": token.value }}
        aria-hidden="true"
      />
      <Text as="span" className="token-mono token-mono-flex">
        {token.name}
      </Text>
      <Text as="span" className="token-mono token-mono-muted">
        {token.label}
      </Text>
    </Flex>
  ))}
</DemoCard>

## Standard format

The [W3C Design Tokens Community Group](https://www.w3.org/community/design-tokens/) is standardizing a JSON format for design tokens. Aligning to it makes your tokens portable between tools like Tokens Studio, Style Dictionary, and others.

None of this tooling is required to start. Plain CSS custom properties are tokens too. Declare them in `:root` (your HTML) and reference them directly, with no build step:

```css
:root {
  --color-text-default: #111827;
  --color-interactive-primary: #2563eb;
  --spacing-component-padding-sm: 8px;
}
```

This covers a single-platform, single-brand system fine. The W3C format and a build step earn their keep once you need the same tokens on iOS or Android, or supported JavaScript files. At that point one source of truth gets transformed into each platform's native format instead of hand-maintained in parallel.

```json
{
  "color": {
    "interactive": {
      "primary": {
        "$value": "{global.color.blue.600}",
        "$type": "color",
        "$description": "Primary action color"
      }
    }
  }
}
```

## Token levels

Teams often organize tokens into three categories:

- **Primitives** — basic values, such as a hex code.

- **Semantics** — names that describe a purpose, such as "action-color".

- **Components** — values for a specific item, such as a button.
