## The system's role

A design system can't guarantee accessible products. That depends on how components are used in context, but it can dramatically raise the floor by making accessible patterns the default.

What the system should own:

- Focus management and visible focus styles on all interactive elements
- Color token pairings that pass contrast in their intended combinations
- Semantic HTML choices in component markup
- ARIA patterns only if necessary
- Motion reduction support via `prefers-reduced-motion`

### Focus styles

Never remove focus outlines without replacing them. The default browser focus ring is functional but many systems replace it with a branded, clearly visible ring:

```css
/* System-level focus reset */
:focus-visible {
  outline: 2px solid var(--color-interactive-primary);
  outline-offset: 2px;
  border-radius: 2px;
}
```

### Motion

```css
@media (prefers-reduced-motion: reduce) {
  *,
  *::before,
  *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}
```

### Skip links

Every page that includes a navigation sidebar needs a "Skip to main content" link as the first focusable element:

```html
<a href="#main-content" class="sr-only focus:not-sr-only">
  Skip to main content
</a>
```

## Testing strategy

Automated tools catch a good amount of accessibility issues, but not all. Manual testing is just as important if not more:

1. **Automated** — axe-core in CI (via jest-axe or Playwright), catching contrast failures, missing labels, and structure issues
2. **Keyboard testing** — navigate every component with Tab, Shift+Tab, Enter, Space, Escape, and arrow keys
3. **Screen reader testing** — VoiceOver on macOS, NVDA + Chrome on Windows. Create a cadence on audits.
4. **Zoom testing** — test at 200% and 400% zoom. Content should reflow, not overflow.

<Callout type="info">
  Add axe-core to your Storybook or component test suite. A failure caught at
  the system level fixes every product that consumes that component which is a
  much better return than catching it in individual product code reviews.
</Callout>
